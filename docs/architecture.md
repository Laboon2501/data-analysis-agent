# Architecture

This project is a structured data-analysis agent built around LangGraph,
Pydantic state, guarded tools, explicit job execution, and artifact references.
It avoids hidden ReAct loops, giant agent classes, and one dispatcher that exposes
all tools to every node.

## LangGraph Workflow Overview

```text
FastAPI / scripts
  -> app.harness builds AgentState and routes intent
  -> app.workers.InMemoryJobRunner selects one graph
  -> graph nodes mutate structured AgentState
  -> events and artifacts are written through store interfaces
  -> API exposes job status, event list, SSE, and artifact reads
```

The default runner is synchronous and in-memory so tests remain deterministic.
Redis, Postgres, Celery, real LLM providers, and MCP transports are optional
adapters or manual integration paths.

## Datasource Registry

The memory API keeps a small process-local datasource registry. A datasource
record contains `datasource_id`, name, kind, masked URL or SQLite path, status,
created time, last profiled time, and schema hash. Real connection-string
passwords are not returned to the frontend.

The registry can be seeded from `DATA_ANALYSIS_AGENT_DATASOURCE_URL` /
`DATA_ANALYSIS_AGENT_DATASOURCE_ID`, and it automatically registers
`demo/ecommerce_demo.sqlite` as `ecommerce-demo-sqlite` when that file exists.
File datasources use the same registry. CSV, Excel xlsx, and optional Parquet
inputs are converted to internal SQLite tables, then queried through the
existing `SQLAlchemyDataSource`, SQLGuard, Context Manager, Direct Analysis,
Open Exploration, and export paths. Registry responses expose only safe file
metadata such as basename, table name, row count, columns, and file size.
Sessions can select a current datasource through:

```text
POST /sessions/{session_id}/datasource
GET /sessions/{session_id}/datasource
```

Analysis, exploration, and profiling jobs use the session datasource. If only
one datasource exists, the memory runner can auto-select it. If none exists, or
if multiple exist without a session selection, the runner returns a clarification
message instead of executing SQL. `POST /datasources/{datasource_id}/profile`
submits the Context Manager graph and updates the registry with `schema_hash`
and `last_profiled_at` when profiling succeeds.

## Context Manager

The Context Manager graph converts datasource metadata into `DatabaseProfile`.

```text
read_schema
-> sample_tables
-> infer_table_roles
-> infer_field_semantics
-> infer_relationships
-> generate_candidate_metrics
-> generate_candidate_dimensions
-> detect_ambiguity
-> save_profile_cache
```

The profile stores tables, columns, row counts, samples, table roles, field
semantics, relationships, candidate metrics, candidate dimensions, ambiguous
fields, schema hash, and profile version. Downstream graphs read the profile
instead of stuffing schema reasoning into chat history.

## Direct Analysis

Direct analysis handles clear user questions.

```text
route
-> ensure_database_profile
-> retrieve_similar_cases
-> interpret_question
-> make_analysis_plan
-> draft_sql
-> validate_sql
-> risk_check_sql
-> execute_sql
-> check_result
-> repair_sql_if_needed
-> decide_chart
-> generate_chart_artifact
-> generate_insight
-> build_analysis_package
-> final_response
```

SQL drafting can be rule-based or opt-in LLM-based. Every draft still goes
through validation and `SQLGuard` before execution. Chart generation writes a
lightweight artifact and only passes `artifact_ref` through events/state.

## Schema QA / Data Inspection

Schema QA handles questions such as "有哪些字段", "这个表有哪些列", "哪些字段
可以作为指标", and "这张表有什么可以分析的吗". It is a separate controlled
LangGraph path:

```text
ensure_database_profile
-> answer_schema_question
```

The graph reads only `DatabaseProfile` and bounded sample metadata generated by
the Context Manager. It does not call `query_data`, does not generate aggregate
SQL, and does not expose uploaded file bodies or artifact bodies in chat
history. When `real_llm` or `fake_llm` is enabled, the LLM may polish the
Chinese answer from a bounded schema summary, but it receives no tools and its
referenced fields must come from the profile.

The original `Zafer-Liu/Data-Analysis-Agent` experience used a broader LLM
harness with automatic tool choice and dispatcher-style execution. This rewrite
keeps natural-language understanding in explicit LangGraph nodes instead:
router, planner, SQL drafter, insight writer, no-tools chat responder, and
schema QA writer are narrow, observable steps, while tool access remains
node-scoped through the registry.

## Open Exploration

Open exploration starts from the `DatabaseProfile` and creates a small set of
candidate analysis topics.

```text
route
-> ensure_database_profile
-> generate_analysis_map
-> rank_topics
-> optional_human_confirm
-> run_top_n_analyses
-> summarize_findings
-> build_exploration_package
-> final_response
```

The current implementation is deterministic and rule-based. The human-confirm
step is represented with structured `HumanRequest` data and can be skipped in
tests.

## Report Export

Report export always separates outline planning from file generation.

```text
analysis_package
-> generate_outline
-> human_confirm
-> export_file
-> return_artifact
```

The first pass creates an outline and stops for confirmation. Confirm commands
such as `report_confirm`, `ppt_confirm`, `excel_confirm`, and
`dashboard_confirm` resume through the fast-path, reuse the existing
`AnalysisPackage` and `ReportOutline`, and do not rerun analysis or regenerate
the outline.

## Artifact, SSE, Worker, Persistence

Runtime configuration is centralized in `app/config.py`. `AppConfig` reads the
runner backend, Redis URL, Celery broker/result backend, checkpoint URL, artifact
directory, datasource URL, session store settings, and LLM placeholders from
environment variables. Safe defaults keep the app on the memory backend.

`SessionStore` is separate from checkpoint and event persistence:

- Session Store saves user-visible history: session records, chat messages, job
  summaries, datasource selection, LLM mode/nodes, and artifact refs.
- CheckpointStore saves workflow state snapshots for job recovery and
  interrupt/resume behavior.
- EventStore saves structured progress events for job event lists and SSE.
- ArtifactStore saves chart/report/Excel/PPT/dashboard content and metadata.

Session history must not contain artifact bodies, chart HTML, file bytes, API
keys, or long internal workflow state. It keeps compact assistant summaries and
`artifact:<id>` references that the frontend can resolve through the artifact
API.

The default `SessionStore` is in-memory. `SQLAlchemySessionStore` can persist the
same user-visible records to SQLite and is structured so a Postgres-compatible
URL can be supported later. Retention is manual or write-time only: stores can
trim old messages by `max_messages`, and `POST /sessions/cleanup` can delete
sessions older than a configured TTL while preserving artifact files.

Artifacts are stored behind `ArtifactStore` implementations. Chart specs,
dashboard specs, reports, Excel workbooks, and PPTX files are retrieved through
the artifact API:

```text
GET /artifacts/{artifact_id}
GET /artifacts/{artifact_id}/content
```

`DashboardSpec` is a lightweight frontend rendering contract saved as a JSON
artifact. It references chart artifacts through `chart_artifact_ref` and keeps
large chart/table/report bodies out of events, final responses, and session
history. The static Web UI renders metric, chart, table, and insight widgets by
reading artifact metadata/content through the same API.

Events use `AgentEvent` and are available as both list and SSE stream:

```text
GET /jobs/{job_id}/events
GET /jobs/{job_id}/events/stream
```

SSE payloads omit known large fields such as rendered chart HTML, file bytes,
binary blobs, and data URLs. The default worker backend is
`InMemoryJobRunner`. The optional Celery backend submits `run_agent_job` tasks
and uses shared cache/event/checkpoint/artifact stores for job status and
progress. In Celery mode, API and worker should share the same file artifact
directory. Redis event storage supports finite listing and polling stream reads
for SSE. Redis, Postgres, file artifact storage, and Celery are not required by
default tests.

## LLM Strategy Fallback

LLM usage is optional and node-scoped. Supported rollout aliases include
`router`, `planner`, `sql_drafter`, and `insight_writer`. Each LLM node loads a
narrow prompt from `prompts/`, expects JSON output, validates that JSON, and
falls back to rule strategy on provider errors, timeouts, or invalid JSON.

LLM observability events include:

- `llm_start`
- `llm_end`
- `llm_error`
- `llm_json_invalid`
- `llm_fallback`

Diagnostic metadata is structured and truncated. API keys and full sensitive
prompts are not stored in state or events.

## MCP Adapter

MCP tools are configured through `mcp/config.py`, connected by the stdio/SSE
transport layer in `mcp/transport.py`, managed by `MCPManager`, and converted
into ToolRegistry entries by `mcp/adapter.py`. Tool names use:

```text
mcp__{server_id}__{raw_tool_name}
```

stdio commands are allowlisted and run with `shell=False`. SSE configuration is
manual and opt-in. MCP tools register into the `mcp` category and still obey
per-node ToolRegistry permissions. No LLM receives automatic access to all MCP
tools.

`examples/mcp/demo_mcp_server.py` is a bundled local stdio sample used only for
manual smoke validation. It exposes read-only demo metadata tools,
`list_demo_tables` and `describe_demo_table`, and validates the same
transport -> `MCPManager` -> ToolRegistry path as any external MCP server. The
sample is disabled in `scripts/mcp.example.json` by default and must be enabled
explicitly in a local copy.
